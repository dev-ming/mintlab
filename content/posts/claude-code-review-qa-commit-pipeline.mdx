## 배경

전에 만들어둔 [프로젝트 전용 에이전트 팀](프로젝트%20전용%20QA%20에이전트%20팀%20만들기.md)은 두 명이었다.

- **app-task** (구현) — 일감 받아 원인 분석 → 수정 → 자체 typecheck
- **app-qa** (검증) — 변경분에 typecheck/lint/test/E2E를 *돌려서* PASS/FAIL

쓰다 보니 빈자리가 보였다. app-qa는 **"명령을 실행해 깨졌나 확인"**한다. 타입이 맞고 테스트가 통과하면 PASS다. 그런데 **"이 코드가 논리적으로 맞나, 엣지케이스를 빠뜨렸나, 더 깊은 곳에서 고쳤어야 하나"** 같은 건 명령 실행으로 안 잡힌다. 타입은 통과하는데 무한 루프가 있을 수 있고, 테스트는 초록인데 비로그인 분기를 놓쳤을 수 있다.

즉 **검증(verification)과 리뷰(review)는 다른 일**이다. 그래서 세 번째 팀원, **코드 리뷰 전용 에이전트**를 추가했다.

---

## "/code-review 명령 있지 않나?" — 빌트인 vs 커스텀

Claude Code엔 빌트인 `/code-review` 스킬이 있다. high-effort recall 리뷰 — diff를 모아 여러 각도(angle)로 결함 후보를 찾고, 검증해서, 심각도순 findings로 뱉는다. 잘 만들어져 있다.

그런데 빌트인은 **범용**이다. 우리 레포의 규칙 — "문자열 하드코딩 금지, 무조건 i18n(en/es/pt)", "디자인 토큰만, 임의 hex/px 금지", "응답은 `{ ok, data }` 엔벨로프", "repo/transaction 패턴" — 을 모른다.

그래서 결론: **빌트인 `/code-review`의 방법론을 그대로 가져오되, 우리 레포 규칙을 아는 커스텀 서브에이전트로 박제**하자. 그러면 파이프라인에서 자동으로 같은 고강도 리뷰가 우리 컨벤션까지 체크하며 돈다.

## app-review 에이전트

`.claude/agents/app-review.md` 하나를 추가했다. 핵심 설계:

- **model: sonnet** — 리뷰는 추론이라 haiku(검증용)보다 한 단계 위. (까다로우면 그때만 opus)
- **recall 편향** — 이 단계에선 진짜 버그를 놓치는 게 오탐보다 나쁘다. 애매하면 일단 surface.
- **다각도 탐색** — 빌트인 `/code-review`의 앵글을 그대로:
  - 정합성: 라인 스캔 / 제거된 동작 / 교차 파일(호출처·콜리)
  - 정리: 재사용(이미 있는 헬퍼 재구현?) / 단순화 / 효율
  - 고도(altitude): 공용 인프라에 특수 케이스 덧댄 임시방편인가
- **프로젝트 규칙 체크** — i18n 하드코딩, 디자인 토큰, 엔벨로프, 공용 컴포넌트 변경 시 전 사용처 회귀, 신규 플로우 E2E 누락
- **검증 후 surface** — "추측이라" 버리지 말고, 코드로 반증 가능할 때만 REFUTED
- **심각도** — must-fix / should-fix / nit. must-fix 하나라도 있으면 `CHANGES_REQUESTED`
- **정지선** — 코드 수정·커밋 안 함. 문제와 고칠 방향만 보고

app-qa와 분담도 명시했다: **qa = 명령 실행 검증, review = 코드 추론.** 같은 일 반복 금지.

## 새 파이프라인: 구현 → 리뷰 → QA → 커밋

기존 "구현 → QA → 사용자 커밋" 사이에 리뷰를 끼웠다.

1. (필요시) `Explore`로 원인 위치 조사
2. `app-task`로 구현
3. **`app-review`로 변경분만 코드 리뷰**
4. `CHANGES_REQUESTED`(must-fix 존재)면 → `app-task`로 되돌려 수정 → 다시 리뷰
5. `APPROVE`면 → `app-qa`로 검증
6. QA FAIL이면 → 재수정 → 리뷰/QA 다시
7. **리뷰 APPROVE + QA PASS면** → 커밋(사용자 승인). Jira 진행중+댓글까지, 완료·푸시는 사람이.

핵심은 **커밋 게이트가 둘**이 됐다는 것. 추론(리뷰)과 실행검증(QA)을 둘 다 통과해야 커밋한다.

## 실제로 써보니 — 리뷰가 진짜로 잡았다

도입하자마자 효과를 봤다. 배포 후 청크 로드 에러를 자동 새로고침으로 복구하는 기능을 만들고(별도 글 참고), 그 변경분을 app-review에 태웠더니 `APPROVE`와 함께 **should-fix 하나**를 짚어냈다:

> 자동 새로고침 누적 카운터가 한도(3회)에 도달하면 에러 화면(global-error)으로 빠지는데, 그 화면에선 카운터를 리셋하는 컴포넌트가 마운트되지 않는다. 그래서 사용자가 수동 버튼으로 새 빌드에 진입해도 **같은 탭에서 다음 배포 땐 카운터가 이미 3이라 자동 새로고침이 0회로 시작**된다. → 수동 버튼 클릭 시 카운터를 함께 리셋해라.

typecheck도 통과하고 테스트도 없는, **순수하게 논리적인 엣지케이스**였다. app-qa로는 절대 안 잡혔을 것이다. 바로 반영하고 다시 리뷰→QA→커밋으로 마무리했다.

## 정리

- **검증 ≠ 리뷰.** qa는 "돌려서 깨졌나", review는 "논리적으로 맞나". 분리하니 각자 잘한다.
- 빌트인 `/code-review`의 고강도 방법론을 **커스텀 에이전트로 박제**하면, 우리 레포 규칙까지 아는 리뷰어가 파이프라인에 상주한다.
- 커밋 게이트를 **리뷰 + QA 이중**으로 두니, 타입은 통과하지만 논리가 틀린 변경을 커밋 전에 거른다.
- 에이전트는 `.claude/agents/*.md` 파일 하나면 끝. 모델·정지선·체크리스트를 프롬프트로 박아두면 매번 같은 기준으로 일한다.

구현하는 사람과 리뷰하는 사람과 QA하는 사람을 분리한다 — 사람 팀에서 당연한 걸, 에이전트 팀에도 그대로 적용했더니 똑같이 잘 굴러간다.
