## 배경

전에 만들어둔 [프로젝트 전용 에이전트 팀](/posts/claude-code-qa-agent-team)은 두 명이었다.

- **kpop-task**: 일감을 받아 원인을 분석하고, 수정하고, 자체 typecheck까지 수행하는 구현 에이전트
- **kpop-qa**: 변경분에 typecheck, lint, test, E2E를 돌려 PASS/FAIL을 판단하는 검증 에이전트

쓰다 보니 빈자리가 보였다. kpop-qa는 **명령을 실행해서 깨졌는지 확인**한다. 타입이 맞고 테스트가 통과하면 PASS다.

그런데 **"이 코드가 논리적으로 맞나", "엣지케이스를 빠뜨렸나", "더 깊은 곳에서 고쳤어야 하나"** 같은 건 명령 실행만으로는 잘 잡히지 않는다. 타입은 통과하는데 무한 루프가 있을 수 있고, 테스트는 초록인데 비로그인 분기를 놓쳤을 수 있다.

즉 **검증(verification)과 리뷰(review)는 다른 일**이다.

그래서 세 번째 팀원, **코드 리뷰 전용 에이전트**를 추가했다.

## "/code-review 명령 있지 않나?" — 빌트인 vs 커스텀

Claude Code에는 빌트인 `/code-review` 스킬이 있다. high-effort recall 리뷰다. diff를 모아 여러 각도에서 결함 후보를 찾고, 검증하고, 심각도순 findings로 정리한다. 잘 만들어져 있다.

그런데 빌트인은 **범용**이다.

우리 레포의 규칙은 모른다.

- 문자열 하드코딩 금지, 무조건 i18n(en/es/pt)
- 디자인 토큰만 사용, 임의 hex/px 금지
- API 응답은 `{ ok, data }` 엔벨로프
- repo/transaction 패턴 유지
- 공용 컴포넌트 변경 시 모든 사용처 회귀 확인

그래서 결론은 이거였다.

> 빌트인 `/code-review`의 방법론은 가져오되, 우리 레포 규칙을 아는 커스텀 서브에이전트로 박제하자.

그러면 파이프라인에서 자동으로 같은 강도의 리뷰가 돌고, 범용 리뷰에 프로젝트 컨벤션 체크까지 얹을 수 있다.

## kpop-review 에이전트

`.claude/agents/kpop-review.md` 파일을 하나 추가했다.

핵심 설계는 이렇다.

- **model: sonnet**
  - 리뷰는 추론이 핵심이라 검증용 haiku보다 한 단계 위 모델을 사용한다.
  - 정말 까다로운 변경이면 그때만 opus를 고려한다.
- **recall 편향**
  - 이 단계에서는 진짜 버그를 놓치는 게 오탐보다 나쁘다.
  - 애매하면 일단 surface한다.
- **다각도 탐색**
  - 빌트인 `/code-review`의 앵글을 가져온다.
  - 정합성: 라인 스캔, 제거된 동작, 교차 파일, 호출처와 콜리 확인
  - 정리: 이미 있는 헬퍼 재구현 여부, 단순화 가능성, 효율
  - 고도: 공용 인프라에 특수 케이스를 덧댄 임시방편인지 확인
- **프로젝트 규칙 체크**
  - i18n 하드코딩
  - 디자인 토큰 위반
  - 응답 엔벨로프 위반
  - 공용 컴포넌트 변경 시 사용처 회귀
  - 신규 플로우 E2E 누락
- **검증 후 surface**
  - "추측이라서" 버리지 않는다.
  - 코드로 반증 가능할 때만 REFUTED 처리한다.
- **심각도**
  - must-fix / should-fix / nit
  - must-fix가 하나라도 있으면 `CHANGES_REQUESTED`
- **정지선**
  - 코드 수정 금지
  - 커밋 금지
  - 문제와 고칠 방향만 보고

kpop-qa와의 분담도 명확히 적었다.

> qa = 명령 실행 검증  
> review = 코드 추론

같은 일을 반복하지 않게 역할을 나눴다.

## 새 파이프라인: 구현 → 리뷰 → QA → 커밋

기존에는 대략 이런 흐름이었다.

```text
구현 → QA → 사용자 확인 → 커밋
```

여기에 리뷰 단계를 끼웠다.

```text
구현 → 코드 리뷰 → QA → 사용자 확인 → 커밋
```

실제 흐름은 이렇게 정리했다.

1. 필요하면 `Explore`로 원인 위치를 조사한다.
2. `kpop-task`가 구현한다.
3. **`kpop-review`가 변경분만 코드 리뷰한다.**
4. `CHANGES_REQUESTED`가 나오면 `kpop-task`로 되돌려 수정한다.
5. 다시 리뷰한다.
6. `APPROVE`가 나오면 `kpop-qa`가 검증한다.
7. QA FAIL이면 재수정하고, 리뷰와 QA를 다시 돈다.
8. **리뷰 APPROVE + QA PASS**가 되면 커밋한다.

커밋 게이트가 둘이 된 셈이다.

- 리뷰: 논리와 설계 추론
- QA: 명령 실행 검증

둘 다 통과해야 커밋한다.

## 실제로 써보니 — 리뷰가 진짜로 잡았다

도입하자마자 효과를 봤다.

배포 후 청크 로드 에러를 자동 새로고침으로 복구하는 기능을 만들고, 그 변경분을 kpop-review에 태웠다. 그러자 `APPROVE`와 함께 **should-fix 하나**를 짚어냈다.

요지는 이랬다.

> 자동 새로고침 누적 카운터가 한도(3회)에 도달하면 에러 화면(global-error)으로 빠진다. 그런데 그 화면에서는 카운터를 리셋하는 컴포넌트가 마운트되지 않는다. 그래서 사용자가 수동 버튼으로 새 빌드에 진입해도, 같은 탭에서 다음 배포 때는 카운터가 이미 3이라 자동 새로고침이 0회로 시작된다. 수동 버튼 클릭 시 카운터를 함께 리셋해야 한다.

typecheck는 통과했다. 테스트도 없었다.

이건 **순수하게 논리적인 엣지케이스**였다. kpop-qa만으로는 잡기 어려운 문제였다.

바로 반영하고 다시 리뷰 → QA → 커밋으로 마무리했다.

## 배운 점

- **검증과 리뷰는 다르다.**
  - QA는 "돌려서 깨졌나"를 본다.
  - 리뷰는 "논리적으로 맞나"를 본다.
- **빌트인 리뷰 방법론은 좋지만, 레포 규칙은 커스텀해야 한다.**
  - 범용 리뷰에 프로젝트 컨벤션을 얹으면 실전성이 올라간다.
- **커밋 게이트는 둘로 나누는 게 좋다.**
  - 리뷰 APPROVE
  - QA PASS
- **서브에이전트는 역할과 정지선이 중요하다.**
  - 리뷰어가 직접 수정하거나 커밋하기 시작하면 파이프라인이 흐려진다.
  - 리뷰어는 문제와 방향만 제시해야 한다.

## 정리

검증은 리뷰가 아니다.

kpop-qa는 명령을 실행해서 깨졌는지 확인한다. kpop-review는 코드와 diff를 읽고 논리적 결함, 빠진 분기, 프로젝트 규칙 위반을 찾는다.

사람 팀에서 구현자, 리뷰어, QA를 나누는 건 당연하다. 에이전트 팀에도 똑같이 적용했다.

```text
kpop-task  →  kpop-review  →  kpop-qa  →  commit
구현          리뷰             검증        기록
```

이 구조를 넣고 나니, 타입은 통과하지만 논리가 틀린 변경을 커밋 전에 한 번 더 걸러낼 수 있었다.

에이전트는 `.claude/agents/*.md` 파일 하나면 충분하다. 모델, 체크리스트, 심각도 기준, 정지선을 프롬프트로 박아두면 매번 같은 기준으로 일한다.

구현하는 사람과 리뷰하는 사람과 QA하는 사람을 분리한다.

사람 팀에서 당연한 걸 에이전트 팀에도 그대로 적용했더니, 꽤 잘 굴러간다.
