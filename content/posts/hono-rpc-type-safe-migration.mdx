Hono로 API를 짤 때, 서버 라우트의 타입을 그대로 프론트로 가져와 코드 생성 없이 end-to-end 타입 안정성을 얻는 방법과, 기존 `fetch` 코드를 점진적으로 옮기는 전략을 정리한 글.

---

## 0. 문제 — 손으로 쓴 fetch는 서버와 따로 논다

프론트에서 API를 부를 때 흔한 방식:

```ts
const res = await fetch(`/api/users/${id}`);
const json = (await res.json()) as { user: User }; // 수동 단언
```

여기서 타입(`{ user: User }`)은 개발자가 손으로 적은 약속일 뿐이다. 서버가 응답 모양이나 경로를 바꿔도 이 캐스팅은 그대로 컴파일된다 → 런타임에서야 깨진다. 요청 바디 형태, 쿼리 파라미터, 경로 오타도 마찬가지로 컴파일러가 안 잡아준다.

OpenAPI + 코드 생성으로 해결할 수도 있지만, 생성 단계(빌드 파이프라인, 산출물 관리) 가 늘어난다.

---

## 1. Hono RPC — 서버 타입을 "그대로" 수입한다

Hono는 라우트를 정의하면 그 타입(경로/메서드/입력 스키마/응답)을 뽑아낼 수 있다. 클라이언트는 그 타입으로 `hc`(hono client)를 만들면, 서버를 타입 수준에서 그대로 호출한다.

```ts
// 서버: 라우트 정의에서 타입을 export
export type AppType = typeof routes;

// 클라: 그 타입으로 클라이언트 생성
const client = hc<AppType>(baseUrl);

// 호출 — 경로/바디/응답이 전부 타입드
const res = await client.users[':id'].$get({ param: { id } });
```

경로 오타, 바디 누락, 응답 필드 접근이 컴파일 타임에 검증된다. 코드 생성 단계는 없다 — 타입은 `typeof`로 서버 코드에서 직접 흘러온다.

---

## 2. 구조 — 3개의 조각

```
[서버]  라우트 정의 → export type AppType = typeof routes
            │  (타입만 흐른다. 런타임 코드는 안 넘어감)
            ▼
[타입 브릿지]  서버의 AppType만 재-export 하는 얇은 패키지 (.d.ts)
            │
            ▼
[클라이언트]  const client = hc<AppType>(baseUrl, { fetch: customFetch })
```

왜 중간에 "타입 브릿지"를 두나 (모노레포):
프론트가 백엔드 패키지를 직접 import 하면, 백엔드의 런타임 의존성·환경변수·서버 전용 코드가 프론트의 타입 그래프와 번들로 새어든다. 그래서 서버 타입만 담은 얇은 패키지(런타임 코드 없이 `export type`만, `.d.ts`로 빌드)를 하나 두고, 프론트는 그 패키지의 타입만 소비한다. 프론트↔백 결합을 "타입 한 줄"로 끊는 것.

작은 프로젝트(프론트·백이 한 패키지)면 이 브릿지 없이 바로 `typeof`를 import 해도 된다.

---

## 3. 라우트가 "타입드"가 되는 3가지 조건

라우트를 만든다고 자동으로 클라이언트에서 타입이 잡히지 않는다. 셋을 모두 지켜야 한다.

① 타입 export 대상(레지스트리)에 포함돼야 한다.
`AppType = typeof X` 에서 그 `X`(라우트 트리)에 편입된 라우트만 타입에 들어간다. 별도로 `app.route(...)` 붙였지만 `X` 밖에 있는 라우트는 클라이언트 타입에 안 뜬다.

② 메서드 체이닝으로 정의해야 한다.
Hono RPC는 체이닝된 반환 타입에서 경로를 추론한다.

```ts
// ✅ 체이닝 — 타입 보존
const routes = new Hono()
  .get('/users', handler)
  .post('/users', validator('json', schema), handler);

// ❌ 끊으면 타입 유실
const app = new Hono();
app.get('/users', handler);   // 이 경로는 클라 타입에 안 잡힘
app.post('/users', ...);
```

③ 입력은 스키마 검증기(validator)로 받는다.
`validator('json', zodSchema)` / `validator('query', ...)` 로 검증한 입력이 클라이언트의 `$post({ json })` / `$get({ query })` 입력 타입의 출처가 된다. 검증기 없이 `c.req` 로 직접 파싱하면 클라 입력 타입이 비어버린다.

---

## 4. 새 타입드 라우트 추가하기

1. 라우트 파일을 체이닝으로 작성 — 입력은 `validator` + 스키마.
2. 타입 export 트리(레지스트리)에 그 라우트를 `.route(...)`로 편입.
3. 타입 브릿지 패키지를 다시 빌드해 `.d.ts`를 갱신 — 안 하면 프론트가 옛 타입을 본다.
4. 프론트에서 `client.xxx.$get/$post(...)` 로 소비.

```ts
// 경로에 하이픈 등 식별자로 못 쓰는 문자가 있으면 브래킷 접근
client.some['kebab-path'].$get({ query: { locale } });
```

## 5. 기존 fetch에서 점진 전환 (hybrid가 정상)

한 번에 다 못 바꾼다. RPC 클라이언트와 기존 fetch를 공존시키며 라우트 단위로 옮기는 게 현실적이다.

- 큰 라우트 파일(도메인 뒤섞임)은 먼저 도메인별로 쪼개 타입 export 트리에 편입(§3-①)해야 타입드가 된다. 이게 전환의 가장 큰 실제 작업.
- 프론트 쪽은 `fetch(url)` + 수동 캐스팅을 `client.…$get/$post` 로 교체.
- 응답 래퍼(envelope) 규약(`{ ok, data }` 등)은 유지되므로, 수동 파싱 대신 타입드 `res.json()` 결과에서 꺼내 쓴다.

---

## 6. 인증은 "커스텀 fetch" 한 곳으로 모은다

`hc`는 내부에서 쓸 `fetch`를 주입받는다.

```ts
const client = hc<AppType>(baseUrl, { fetch: customFetch });
```

이 `customFetch` 하나에 인증을 몰아넣으면 RPC든 직접 fetch든 동일하게 처리된다:

- `Authorization: Bearer <token>` 자동 주입
- `401` 이면 refresh로 새 토큰 받아 원요청 자동 재시도(동시 401은 single-flight로 refresh 1회만)
- refresh도 실패하면 세션 정리 + 로그인 유도
- 비로그인 상태에서 인증 전용 경로 호출은 조용히 빈 상태로 처리

핵심: 인증/토큰 갱신 로직이 호출부마다 흩어지지 않고 fetch 레이어 한 곳에 있다.

---

## 7. 함정 (Gotchas)

- 체이닝 필수 — 중간에 끊으면 그 라우트 타입이 사라진다(§3-②).
- 레지스트리 누락 — 라우트만 만들고 타입 export 트리에 안 넣으면 클라에서 안 잡힌다(§3-①).
- 타입 브릿지 재빌드 — 서버 라우트 추가 후 `.d.ts`를 다시 안 뽑으면 프론트가 옛 타입을 본다.
- 타입 전용 패키지 유지 — 브릿지에 런타임 코드가 섞이면 프론트로 서버 의존성이 새어든다. `export type`만.
- 식별자 불가 경로 — 하이픈 등은 점 접근 대신 `['...']` 브래킷.

---

## 한 줄 요약

Hono RPC는 서버 라우트의 `typeof` 타입을 클라이언트가 그대로 수입해서, 코드 생성 없이 경로·입력·응답을 컴파일 타임에 맞춘다. 관건은 셋 — (1) 타입 export 트리에 편입, (2) 메서드 체이닝, (3) validator로 입력 정의 — 이고, 모노레포면 타입 전용 브릿지 패키지로 프론트↔백 결합을 끊는다. 전환은 fetch와 공존시키며 라우트 단위로 점진적으로.
