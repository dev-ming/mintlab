JWT 기반 로그인에서 access token과 refresh token이 각각 무엇이고, 어떤 방향으로 동작하는지 정리한 글.

---

## 0. 토큰 방식 세션이란

- 옛날(stateful): 서버가 로그인 상태를 메모리/DB에 보관하고, 클라는 세션 ID만 들고 다님.
- 토큰 방식(stateless): 서버는 상태를 안 들고, 클라이언트가 매 요청에 "나 로그인됨"을 증명하는 토큰을 가져온다. 서버는 서명만 검증하면 됨.

토큰은 보통 access / refresh 두 종류로 나눠 쓴다.

---

## 1. Access Token — "매번 보여주는 출입증"

- 모든 API 요청에 첨부해 인증을 증명한다 (`Authorization: Bearer <token>` 또는 쿠키).
- 보통 JWT: 토큰 자체에 정보(누구인지 `sub`, 만료 시각 `exp`)가 담기고 서명돼 있어, 서버가 DB 조회 없이 서명만 검증하면 된다(빠름, self-contained).
- 수명이 짧다 (예: 15분~1시간). JWT는 한번 발급되면 만료 전까지 취소가 어렵기 때문에, 탈취되더라도 악용 가능한 시간을 짧게 가져간다.

## 2. Refresh Token — "금고에 둔 재발급용 카드"

- API 요청엔 쓰지 않는다. 오직 access가 만료됐을 때 새 access를 받아오는 용도.
- 수명이 길다 (예: 1일~30일).
- 핵심 차이: 서버가 추적한다. 보통 서버 측 저장소(예: Redis 같은 인메모리 store)에 `토큰 → 사용자ID`로 저장해 둔다. 덕분에:
  - 들어온 refresh가 저장소에 있는지 확인 → 유효성 검증 가능
  - 로그아웃/탈취 시 저장소에서 지우면 즉시 무효화(revoke) — access(JWT)는 못 하는 것

> 참고: refresh 저장소로 Redis를 자주 쓰는 이유 — 메모리 기반이라 빠르고, TTL(만료) 기능이 있어 "N일 뒤 자동 삭제"를 store가 알아서 처리하며, 여러 서버 인스턴스가 같은 store를 공유할 수 있기 때문.

---

## 3. 왜 굳이 둘로 나누나? (트레이드오프)

토큰 하나로만 하면 딜레마가 생긴다:

- 수명을 짧게 → 안전하지만 자주 재로그인(UX 나쁨)
- 수명을 길게 → 편하지만 탈취 시 오래 악용됨(위험)

둘로 분리하면 둘 다 잡는다:

- access는 짧게(안전) + 매 요청용
- refresh는 길게(편의) + 갱신 전용이고, 서버가 취소 가능(안전)

---

## 4. 누가 무엇을 들고 있나 (방향이 핵심)

| 보관 주체 | 가진 것 |
| --- | --- |
| 클라이언트 | access token + refresh token 둘 다 보관 |
| 서버(저장소) | refresh token → 사용자ID (검증용 명단) |

⚠️ 흔한 오해: "access가 만료되면 저장소에서 refresh를 가져온다"는 틀린 설명.

→ 실제로는 클라이언트가 자기가 가진 refresh를 서버에 제시하고, 서버가 저장소와 대조해 검증하는 것이다. (클라가 저장소에서 꺼내는 게 아님)

---

## 5. 동작 흐름 (the dance)

```
[로그인]
  → access(짧음) + refresh(긺) 발급 → 클라가 둘 다 저장

[요청]  access 첨부
  ├ access 유효 → 정상 응답
  └ access 만료(401)
        → 클라가 [보관 중인 refresh]를 서버 갱신 엔드포인트로 전송   (클라 → 서버)
        → 서버가 저장소에서 refresh 검증
              ├ 유효 → 새 access + 새 refresh 발급 → 클라가 교체 → 원래 요청 재시도
              └ 무효/만료 → 401 → 재로그인
```

대부분 이 갱신은 사용자 몰래 자동으로 일어나서, 활동 중에는 끊김 없이 쓰다가 refresh까지 만료될 때만 다시 로그인하게 된다.

비유하면: refresh는 지갑 속 멤버십 카드, 서버 저장소는 프런트데스크의 회원 명단이다. 만료된 입장권(access)을 갱신하러 카드를 제시하면, 데스크가 명단과 대조한 뒤 새 입장권을 내준다. 데스크가 명단에서 지우면(취소) 그 카드는 즉시 무효가 된다.

---

## 6. 보안 포인트

- 저장 위치: 토큰을 httpOnly 쿠키에 두면 JS가 읽을 수 없어 XSS에 안전하다. `localStorage`는 다루기 편하지만 XSS에 취약.
- 회전(rotation): refresh를 한 번 쓰면 폐기하고 새 refresh를 발급하는 방식. 탈취된 refresh가 재사용되면 감지/무효화할 수 있다.
- revoke(취소): 로그아웃 시 refresh를 저장소에서 삭제. 필요하면 만료 전 access도 별도 블록리스트로 무효화할 수 있다.

---

## 한 줄 요약

access = 짧고 매 요청에 쓰는 출입증(빠른 검증, 탈취 피해 최소화), refresh = 길고 안전하게 보관하는 재발급 카드(서버가 취소 가능, 자동 갱신으로 무중단). 둘을 나눠 보안과 UX를 동시에 챙기는 게 핵심이다.
