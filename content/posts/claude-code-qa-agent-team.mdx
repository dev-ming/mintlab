## 배경

백엔드 개발자가 팀을 떠나면서, 나는 프론트엔드와 백엔드를 혼자 맡게 됐다. 일감은 줄지 않는데 손은 하나다. "일감을 던지면 알아서 조사하고, 고치고, 테스트까지 돌려주는 무언가"가 있으면 좋겠다고 생각했다.

Claude Code에는 **서브에이전트(subagent)** 기능이 있다. 메인 세션이 PM처럼 여러 일꾼 에이전트에게 작업을 나눠 맡기고, 결과를 취합하는 구조를 만들 수 있다. 나는 이걸로 우리 프로젝트(K-pop 팬 플랫폼) 전용 **개발 팀**을 꾸렸다.

핵심 원칙은 하나였다. **"AI가 끝까지 알아서"가 아니라, "AI가 검수 직전까지, 최종 결정은 내가"**.

---

## 결과 먼저 보기

이제 나는 이렇게 말하면 된다.

```
KPOP-303 처리해줘
```

그러면 백그라운드에서:

1. **구현 에이전트**가 코드베이스를 조사 → 원인 분석 → 수정 → 타입체크
2. 끝나자마자 **QA 에이전트**가 그 변경분에 타입체크·린트·E2E 테스트를 독립적으로 실행
3. QA가 실패하면 구현 에이전트로 되돌려 재수정
4. 통과하면 메인 세션이 **diff + QA 리포트 + 제안 커밋 메시지 + Jira 댓글 초안**을 모아 나에게 제출

커밋·푸시·Jira 완료 처리는 **하지 않는다**. 내가 리포트를 보고 직접 결정한다. 여러 일감을 한 번에 주면 각자 병렬로 돌아간다.

---

## 전체 아키텍처

```
사용자 → 메인 세션 (PM)
            │
            ├─ Explore 에이전트 ──→ 코드베이스 조사 (읽기 전용)
            │
            ├─ kpop-task 에이전트 ─→ 원인 분석 → 수정 → 자체 타입체크
            │        │
            │        ▼ (구현 직후)
            ├─ kpop-qa 에이전트 ──→ 타입체크 / 린트 / 단위테스트 / Playwright E2E
            │                        독립 검증, 코드 수정 안 함, PASS/FAIL만
            │
            └─ 결과 취합 → 사용자에게 제출 → 사용자가 최종 검수·커밋·완료
```

에이전트 정의는 레포의 `.claude/agents/` 에 마크다운 파일로 둔다. 한 번 만들어두면 팀원도 git으로 공유받고, 다음 세션부터 자동으로 인식된다.

## 구성원

| 역할 | 에이전트 | 모델 | 책임 | 정지선 |
|---|---|---|---|---|
| 조사 | Explore (내장) | — | 코드 위치·원인 탐색 | 읽기 전용 |
| 구현 | kpop-task | sonnet | 분석 → 수정 → 타입체크 | 커밋·푸시·Jira 안 함 |
| QA | kpop-qa | haiku | 변경분 독립 검증 | 수정·커밋 안 함, 보고만 |
| PM | 메인 세션 | — | 파이프라인 연결·취합 | — |
| 최종 검수 | **나(사람)** | — | diff·QA 검토 후 커밋/완료 | — |

## 핵심 설계 결정

### 1. "정지선"을 명확히 — 구현 담당과 QA 담당을 분리

서브에이전트는 실행 중에 사람에게 질문할 수 없다(자율 실행 후 결과만 반환). 그래서 **되돌리기 쉬운 작업(코드 수정·테스트)까지만** 하고, 비가역·외부 반영 작업(커밋·푸시·Jira 완료·Confluence)은 절대 하지 않도록 못 박았다.

또 구현한 에이전트가 자기 결과를 채점하면 안 되니까, **QA를 별도 에이전트로 분리**했다. 독립 검증이 목적이다.

### 2. 모델 등급으로 토큰 비용 최적화

모든 에이전트를 비싼 모델로 돌릴 필요는 없다.

- **구현(kpop-task)** → sonnet: 원인 분석·수정에 적당한 추론력 + 합리적 비용
- **검증(kpop-qa)** → haiku: "명령 실행하고 PASS/FAIL 판정"이라 가장 저렴한 모델로 충분

메인 세션 모델을 그대로 물려받게(inherit) 두면 비싼 모델을 따라가므로, **에이전트별로 모델을 명시 고정**했다. 어려운 건은 그때만 "이건 opus로 해줘"로 올린다. 여기에 더해 모든 에이전트에 "필요한 파일·라인만 읽기, 로그·코드 통째 인용 금지, 간결 보고" 같은 토큰 절약 규칙을 넣었다.

### 3. Playwright E2E + dev-login으로 "진짜 동작" 검증

타입체크만으로는 "버튼을 눌렀을 때 실제로 동작하는가"를 보장 못 한다. 그래서 **Playwright E2E**를 도입했다.

가장 큰 걸림돌은 로그인이었다. 우리 서비스는 Firebase 인증을 쓰는데, E2E에서 Firebase OAuth를 흉내 내는 건 까다롭다. 다행히 백엔드에 **개발용 로그인 엔드포인트(`POST /api/v1/auth/dev-login`, production 외에서만 동작)**가 있었다. 이걸로 Firebase 없이 토큰을 발급받아 sessionStorage에 심는 **인증 픽스처**를 만들었더니, 로그인이 필요한 페이지도 테스트할 수 있게 됐다.

> 함정 하나: 로그인 후 화면에 5.6초짜리 인트로 스플래시와 쿠키 동의 배너가 떠서 버튼 클릭을 가로막았다. 둘 다 sessionStorage / localStorage 플래그로 제어되길래, 픽스처에서 미리 그 값을 심어 오버레이를 꺼버렸다.

### 4. QA가 프론트 변경을 감지하면 자동으로 E2E 실행

dev 서버가 상시 떠 있는 환경이라, QA 에이전트는 docker나 API 서버를 새로 띄우려 하지 않고 떠 있는 서버를 그대로 재사용한다(Playwright `reuseExistingServer`). 불필요한 셋업에 시간·토큰을 쓰지 않게 했다.

### 5. 테스트 플로우가 기능과 함께 자동으로 늘어나게

테스트는 한 번 만들고 끝이 아니다. 기능이 추가될 때마다 누군가 "이것도 테스트 만들어야지"를 기억해야 한다면, 결국 빠진다. 그래서 이걸 **프로세스 규칙으로 박았다.**

- **구현 에이전트(kpop-task)**: 새 사용자 플로우/페이지/기능을 만들면, 그 자리에서 `e2e/`에 대응 테스트를 같이 작성하는 것을 기본 산출물로 한다.
- **QA 에이전트(kpop-qa)**: 신규 플로우인데 대응 E2E가 없으면 통과시키지 않고 "커버리지 누락"으로 내려 짚는다.
- **작성 가이드**: `e2e/README.md`에 공개 페이지·로그인 필요·버그 회귀 3종 템플릿을 문서화해서, 사람이든 에이전트든 같은 패턴으로 따라 쓰게 했다.

즉 "기능을 추가하면 테스트도 자동으로 추가되고, 빠지면 QA가 잡는다." 테스트 커버리지가 기능 증가를 따라가지 못해 썩는 흔한 문제를, 사람의 기억력이 아니라 워크플로 규칙으로 막은 것이다.

## 실제 검증 결과

첫 사례로, 바로 직전에 고친 **출석체크 버그(KPOP-290)** 를 회귀 테스트로 만들었다.

> 증상: 로그인 상태인데도 출석 체크인 버튼을 누르면 "로그인 필요" 모달이 떴다.
> 원인: `useCallback` 의존성 배열에 `isLoggedIn`이 빠져, 콜백이 초기값 `false`를 계속 참조하는 stale closure.

E2E 시나리오: dev-login으로 로그인 → 출석 페이지 진입 → 체크인 클릭 → **"Login Required" 모달이 뜨지 않아야 한다**.

```
공개 페이지 스모크 4개 + 출석체크 회귀 1개 = 5개 전부 통과 (8.2초)
```

수정 전 코드였다면 이 테스트는 실패한다. 즉 같은 버그가 다시 들어오면 자동으로 잡힌다.

## 사용 기술

| 구성 요소 | 설명 |
|---|---|
| Claude Code 서브에이전트 | `.claude/agents/*.md` 로 역할별 에이전트 정의 |
| 모델 등급 지정 | 에이전트별 `model:` (sonnet / haiku)로 비용 최적화 |
| Playwright | web-front E2E. dev 서버 자동 기동(`webServer`) |
| dev-login 엔드포인트 | Firebase 우회 테스트 인증 → sessionStorage 주입 픽스처 |
| 병렬 처리 | 일감 간 동시 실행, 일감 내 구현→QA는 순차 |

## 배운 점

- **자율성과 통제의 경계를 코드가 아니라 "정지선"으로 설계한다.** AI에게 끝까지 맡기는 게 아니라, 검수 직전까지만 시키고 비가역 작업은 사람이 결정하게 하니 안심하고 위임할 수 있었다.
- **검증 담당을 분리하면 신뢰가 올라간다.** 만든 사람이 채점하지 않는다는 원칙은 사람 팀이든 AI 팀이든 똑같이 유효하다.
- **모델은 작업 난이도에 맞춰 골라 쓴다.** 기계적인 검증까지 비싼 모델을 쓸 이유가 없다. 역할별 모델 지정만으로 비용이 크게 줄었다.
- **E2E의 진짜 장벽은 인증이다.** 테스트 전용 로그인 우회 경로(dev-login) 하나가 E2E 가능 범위를 통째로 바꿔놓았다.
