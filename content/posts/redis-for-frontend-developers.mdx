Redis는 백엔드 기술이지만, 프론트엔드 개발자가 겪는 여러 현상(로그인 유지, 캐시된 응답, 429 에러 등)의 배경에 있다. "내가 직접 쓰진 않지만 알면 디버깅·소통이 쉬워지는" 관점에서 정리.

---

## 한 줄 정의

Redis = 서버 쪽의 빠른 "단기 기억 메모장" (인메모리 key-value 저장소).

- 데이터를 주로 메모리(RAM)에 둬서 매우 빠름.
- 구조는 단순: key → value (`set` / `get` / `del`).
- TTL(만료): "이 값 N초 뒤 자동 삭제" 가능 → 잠깐 사는 데이터에 최적.
- 여러 서버 인스턴스가 같은 Redis를 공유 → 상태를 한 곳에 모음.

---

## 프론트 스토리지랑 뭐가 다른가? 

| | localStorage / sessionStorage / 쿠키 | Redis |
| --- | --- | --- |
| 위치 | 브라우저(클라이언트) | 서버 |
| 누가 읽나 | 내 브라우저의 JS | 백엔드 코드 |
| 용도 | 토큰 보관, UI 상태 | 세션·캐시·레이트리밋 등 서버 상태 |
| 접근 | FE가 직접 read/write | FE는 직접 못 봄(API 너머) |

→ 즉 둘 다 "key-value 임시 저장"인데, 하나는 내 브라우저, 하나는 서버에 있는 것. 프론트는 Redis를 직접 만지지 않고, API를 통해 간접적으로 그 효과를 본다.

---

## 프론트가 Redis를 "체감"하는 순간들

### 1. 로그인 유지 / 로그아웃
- refresh token이 Redis에 저장돼 세션이 유지됨. → 그래서 새로고침해도 로그인 유지.
- 로그아웃하면 서버가 Redis에서 토큰을 지움 → 모든 기기/탭에서 즉시 로그아웃되는 게 이 덕분.
- 관련 개념: [[토큰 인증 — Access vs Refresh]]

### 2. 빠른 응답 & "왜 옛날 데이터가 보이지?"
- 백엔드가 자주 쓰는 데이터를 Redis에 캐시해두면 API가 빨라짐.
- 단점: 캐시 TTL 동안 stale(오래된) 데이터가 보일 수 있음. → "DB는 바뀌었는데 화면은 그대로"의 흔한 원인. (캐시 무효화 타이밍 이슈)

### 3. 429 Too Many Requests
- 레이트리밋(요청 횟수 제한)을 Redis 카운터로 구현하는 경우가 많음. → 짧은 시간에 너무 많이 호출하면 429.

### 4. 실시간/알림
- pub/sub 기능으로 알림·실시간 기능의 뒤를 받치기도 함.

## 왜 DB(Postgres 등) 대신 Redis?

- DB는 디스크 기반이라 안전·영속적이지만 상대적으로 느림 → 영구 데이터(유저, 주문 등).
- Redis는 메모리 기반이라 빠르지만 휘발 가능 → 잠깐 살아도 되는, 빠른 접근이 필요한 데이터(세션, 캐시, 카운터).
- 그래서 보통 둘을 같이 쓴다: 영구 데이터는 DB, 빠른 임시 데이터는 Redis.

---

## 디버깅할 때 알아두면 좋은 것

- "로그인이 자꾸 풀려요" → Redis가 재시작됐거나(메모리라 날아감), 인스턴스 간 공유가 안 되거나, TTL이 너무 짧을 수 있음.
- "수정했는데 반영이 안 돼요" → Redis 캐시 TTL이 안 지났을 수 있음(서버 캐시 무효화 필요).
- 프론트에서 직접 고칠 수 있는 건 아니지만, "이거 캐시/세션(서버 쪽) 문제 같다"고 백엔드와 정확히 소통할 수 있게 됨.

---

## 한 줄 요약

Redis는 서버의 빠른 임시 메모장. 프론트는 직접 다루진 않지만, 로그인 유지·빠른 응답·stale 데이터·레이트리밋의 배경에 Redis가 있다고 알아두면 현상 이해와 백엔드 소통이 쉬워진다.
