## 증상

배포하고 나면 가끔 이런 일이 생긴다. 사용자가 하단 탭바의 Home을 누르면 404가 뜨거나, 콘솔에 이런 에러가 찍힌다.

```
Uncaught Error: Module 65205 was instantiated because it was required from
module 38116, but the module factory is not available
```

특징이 묘했다.

- 로컬에선 안 뜨고, 배포된 환경에서만 뜬다.
- 항상은 아니고 "종종", 그리고 URL 이동(페이지 전환) 할 때 터진다.
- 새로고침하면 멀쩡해진다.

원인을 모르면 "랜덤 버그"처럼 보이지만, 이 세 가지 단서만으로 범인은 거의 특정된다.

---

## 오진 먼저 (남기는 이유: 같은 길로 새지 말라고)

처음엔 dev 환경 얘기인 줄 알고 Turbopack HMR 캐시 불일치를 의심했다. 하지만 사용자가 "배포 후에만, 로컬은 멀쩡"이라고 정정해주면서 그 가설은 폐기됐다. dev HMR이면 로컬에서 더 자주 나야 한다.

다음 의심은 서비스 워커(PWA). 옛 자산을 캐시해서 주는 거 아니냐. `sw.js`를 열어보니 — 이 SW는 `navigate` 요청만 network-first로 처리하고 JS 청크는 캐시하지 않았다(애초에 `cache.put`이 없었다). 무죄.

교훈: 같은 에러도 "어디서(로컬/배포), 언제(이동 시), 얼마나(간헐)"를 정확히 좁히면 엉뚱한 데 파지 않는다.

---

## 진짜 원인: 배포 스큐(deploy skew)

web dev 환경은 in-house Docker로, Next.js `output: standalone` 빌드를 컨테이너가 직접 서빙한다. (Next 16은 dev/build 둘 다 Turbopack이 기본이라, 산출물 청크 이름이 `turbopack-*`이고 에러도 Turbopack 런타임 형식으로 난다.)

흐름을 그려보면:

1. 사용자가 앱을 연다 → 빌드 A의 청크 그래프를 브라우저가 들고 있다.
2. 그 사이 새 배포 → 컨테이너가 빌드 B로 교체. 빌드 A의 `/_next/static/...` 청크는 사라진다(새 컨테이너엔 빌드 B 청크만 있음).
3. 사용자가 페이지를 이동(client-side navigation)하면, Next 라우터가 그 화면에 필요한 빌드 A의 청크를 lazy-load 시도 → 새 컨테이너엔 없음 → 404 / "module factory is not available".

즉 "배포 전부터 열어둔 세션"이 "배포 경계를 넘어 페이지를 이동"할 때만 터진다. 증상의 세 단서와 정확히 맞는다.

> CDN invalidation 문제가 아니다(여긴 CDN이 관리하는 구조가 아님). 그냥 컨테이너 교체로 옛 청크가 물리적으로 없어지는 것이다. invalidation은 "새로 들어오는 방문자"만 도울 뿐, 이미 열린 세션의 스큐는 못 고친다.

---

## 해결: 청크 에러를 감지해 자동 새로고침

옛 청크가 없어서 나는 에러이므로, 감지하면 전체 새로고침으로 새 빌드를 받아오면 끝이다. 문제는 "그 에러가 어디로 표면화되느냐"가 두 갈래라는 것.

- 비동기/uncaught 경로 — 동적 import 실패는 Promise reject(`unhandledrejection`), 모듈 평가 중 throw는 `window.onerror`.
- React 렌더 경로 — 렌더 중 터지면 React가 가로채 error boundary로 보낸다(window 이벤트로 안 감).

그래서 둘 다 잡아야 한다.

```
lib/chunk-reload.ts          ← 감지(isChunkLoadError) + 가드 새로고침(공용)
app/components/.../ChunkReloader.tsx  ← window error/unhandledrejection 경로
app/global-error.tsx         ← React 렌더 경로(root error boundary)
```

감지는 메시지 정규식으로: `ChunkLoadError`, `Loading chunk … failed`, `module factory is not available`, `was instantiated because it was required from module`, `Failed to fetch dynamically imported module` 등.

---

## 함정: "자동 새로고침"은 무한 루프가 되기 쉽다

여기서부터가 진짜다. 단순히 "에러 나면 reload"는 위험하다.

- 진짜 깨진 빌드(배포 스큐가 아니라 실제 버그)면 → 새로고침해도 또 터짐 → 무한 새로고침.
- sessionStorage가 막힌 환경이면 가드가 안 먹어 고속 폭주.
- 가드가 너무 세면 → 새로고침은 막혔는데 화면엔 "Updating…"만 뜨고 사용자가 갇힘.

그래서 가드를 이렇게 설계했다.

1. 모듈 스코프 `reloadedThisLoad` — 한 번의 페이지 로드당 새로고침 1회만. sessionStorage가 막혀도 폭주를 막는다.
2. sessionStorage 누적 카운터(MAX 3) — 새로고침이 누적 3회를 넘으면 포기(무한 루프 차단).
3. `markChunkLoadHealthy()` — 청크 에러 없이 5초간 살아있으면 정상으로 보고 카운터 리셋(다음 배포 땐 다시 풀로 시도).
4. global-error엔 항상 보이는 수동 버튼 — 자동이 캡에 막혀도 사용자가 직접 빠져나갈 수 있게. (버튼 누르면 카운터 리셋 + 강제 새로고침)

## 코드 리뷰가 잡아준 한 끗

이 변경을 [코드 리뷰 에이전트](코드%20리뷰%20에이전트%20추가하고%20리뷰→QA→커밋%20파이프라인%20만들기.md)에 태웠더니 `APPROVE`와 함께 엣지케이스를 짚었다.

> 카운터가 캡(3)에 도달하면 에러 화면(global-error)으로 빠지는데, 그 화면에선 카운터를 리셋하는 컴포넌트가 마운트되지 않는다. 그래서 사용자가 수동 버튼으로 새 빌드에 진입해도, 같은 탭에서 다음 배포 땐 카운터가 이미 3이라 자동 새로고침이 0회로 시작된다.

→ 수동 버튼 클릭 시 카운터를 함께 리셋하도록 고쳤다. typecheck도 통과하고 테스트도 없는, 순수 논리 엣지였다.

## 검증 — 재배포 없이 확인하기

청크 에러는 재배포해야 재현되니, 로직만 따로 검증했다.

- 격리 프로세스 시뮬: sessionStorage·location.reload를 스텁으로 갈아끼우고 실제 `chunk-reload.ts`를 import → `정상=reload 1회 / 누적 3회 도달=0회(포기) / storage 막힘=폭주 없이 1회`를 모두 확인.
- 브라우저 콘솔 1줄로 end-to-end:
  ```js
  window.dispatchEvent(new ErrorEvent('error', {
    message: 'the module factory is not available'
  }))
  ```
  → 페이지가 자동 새로고침되면 성공. (재시도는 `sessionStorage.removeItem('chunk-reload-attempts')` 후)

## 정리

- 배포 후 "가끔, 이동할 때 404"는 십중팔구 배포 스큐: 옛 세션이 사라진 옛 청크를 lazy-load.
- CDN invalidation 문제가 아니라 빌드 교체로 옛 청크가 없어지는 것 — 자동 새로고침으로 새 빌드를 받게 하는 게 실용적 해법.
- 에러는 window 경로 + React 렌더 경로 둘로 새므로 ChunkReloader + global-error 둘 다 둔다.
- "자동 새로고침"은 무한 루프/폭주/갇힘을 반드시 같이 설계해야 한다(로드당 1회 + N회 캡 + 정상 리셋 + 수동 탈출 버튼).
- 근본 차단을 더 원하면: `deploymentId` 설정, 옛 빌드 정적 에셋 보존(assetPrefix→CDN), HTML 단기 캐시.

"랜덤하게 404 뜬다"는 신고를, 세 가지 단서로 좁혀 배포 스큐로 특정하고, 자동 복구로 사용자가 모르고 지나가게 만든 사례.
