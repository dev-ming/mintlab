다국어(i18n)를 쓰는 웹 서비스에서 "번역 키를 추가했는데 어디론가 사라지는" 문제를 추적하고, 코드와 번역용 스프레드시트 사이의 동기화를 자동화한 기록이다.

---

## 배경: 번역은 어디에 사는가

다국어 서비스에서 번역 문자열은 보통 두 곳에 동시에 존재한다.

- **코드 쪽**: 앱이 실제로 읽는 메시지 파일(`messages/en.json` 같은 로케일별 JSON).
- **사람 쪽**: 번역가가 작업하는 **스프레드시트**(언어별 열을 두고 한 줄에 한 키).

내가 맡은 서비스도 그랬다. 빌드할 때 스프레드시트를 읽어 JSON으로 내려받는 스크립트가 이미 있었다. 번역가가 시트를 채우면, 배포 빌드가 그 내용을 코드로 가져온다. 즉 **스프레드시트가 원본(source of truth)** 이고, 흐름은 한 방향이었다.

```
스프레드시트  ──(빌드 시 download)──▶  messages/*.json
```

---

## 문제: 추가한 키가 사라진다

개발자가 새 화면을 만들면서 번역 키를 하나 추가했다고 하자. 보통 코드의 메시지 파일에 직접 적는다. 그런데 다음 배포 빌드가 돌면, **스프레드시트를 읽어 JSON을 통째로 덮어쓴다.** 시트에는 그 새 키가 없으니, 방금 추가한 키는 그대로 증발한다.

겉으로 드러난 증상은 "로컬라이징을 추가했는데 동기화가 안 된다"였지만, 본질은 **단방향 구조의 구멍**이었다. 코드에서 시트로 거슬러 올라가는 길(export)이 아예 없었던 것이다. 번역가도 새 키가 생긴 걸 알 방법이 없었다.

```
[개발자가 코드에 키 추가]  ✗ 시트로 올라가는 길 없음
        ▼ 다음 빌드의 download가 JSON 덮어씀
[새 키 소멸]
```

---

## 해결: 반대 방향 길을 낸다 (export)

필요한 건 **JSON → 스프레드시트** 방향이었다. 다만 욕심내지 않고 딱 한 가지 동작만 하게 했다.

> 코드 메시지 파일에서 **시트에 아직 없는 키만 골라, 시트에 새 행으로 추가(append)** 한다. 원문(기준 언어) 값만 채우고 나머지 언어 칸은 비워 둔다 — 번역가가 채울 자리니까.

핵심 제약은 **append-only**. 기존 행이나 이미 들어간 번역은 **절대 건드리지 않는다.** 동기화 스크립트가 실수로 번역을 덮어쓰는 건 최악이기 때문이다. 그래서 "추가만 한다, 수정·삭제는 안 한다"를 불변 규칙으로 박았다.

구현은 기존 download 스크립트에 함수를 하나 더 붙이는 형태였다.

- 중첩 JSON을 점 표기(dot-notation) 키로 평탄화한다.
- 시트의 기존 키 집합을 읽어 와 비교한다.
- 차집합(시트에 없는 키)만 새 행으로 append한다.

방향을 시점으로 나눠 정리하면 이렇게 된다.

```
[개발자가 키 추가] → push
   → (CI) EXPORT  : 신규 키를 시트에 append (원문만, 나머지 언어 빈칸)
[번역가가 시트에서 번역 채움]
   → (빌드) IMPORT : 시트 → JSON, 번역 회수
```

- **export = push 때** (새로 만든 자동화)
- **import = 빌드 때** (원래 있던 것)

두 방향이 다른 시점에 돌기 때문에 충돌하지 않고, 루프가 닫힌다.

---

## 자동화: "추가하면 알아서 올라가게"

수동으로 스크립트를 돌릴 수도 있지만, 사람이 기억해야 하는 일은 결국 빠진다. 그래서 **CI에 얹었다.** 메시지 파일이 바뀐 채로 push되면, 파이프라인이 export 스크립트를 자동 실행하도록.

여기서 작은 설계 선택이 있었다. 원래는 기본 브랜치에서만 돌게 잡았는데, "작업하면서 바로바로 올라가면 좋겠다"는 요구에 맞춰 **브랜치 제한을 풀어 어느 브랜치 push든 동작**하게 했다. 대신 트레이드오프를 메모로 남겼다 — 작업 중 브랜치의 실험적 키도 시트에 미리 올라간다(append-only라 번역은 안 망가지지만, 나중에 버린 키가 시트에 남을 수 있음).

자격증명은 시크릿이라 저장소에 두지 않는다. 로컬엔 키 파일이 있지만 그건 버전관리에서 제외돼 있고, CI 서버엔 그 파일이 없다. 그래서 **CI 시크릿에 키를 넣어 두고, 파이프라인이 실행 시점에 그걸 파일로 풀어 쓰게** 했다.

---

## 함정: read는 되는데 write만 막힌다

자동화를 붙이기 전에 로컬에서 더미 키 하나로 테스트했다. 그런데 첫 실행에서 **403 (권한 없음)**.

로그를 보니 흥미로웠다. 스크립트는 **시트를 읽는 데는 성공**했다(기존 키 수백 개를 정상적으로 가져왔다). 그런데 **새 행을 추가하는 순간에만 403**이 났다. 인증 자체는 됐고, 코드의 권한 범위(scope)도 쓰기로 맞춰 둔 상태였다.

원인은 코드가 아니라 **공유 권한**이었다. 서비스 계정에 시트가 **뷰어(읽기 전용)** 로만 공유돼 있었던 것이다. 그래서 다운로드(읽기)는 처음부터 잘 됐고, 이번에 새로 추가한 업로드(쓰기)만 막혔다. 시트를 **편집자**로 공유하자 곧바로 통과했고, "신규 키 1개 추가"가 깔끔하게 찍혔다. 기존 행은 그대로, 한 줄만 늘었다 — append-only도 같이 검증됐다.

이 디버깅에서 건진 교훈: **"인증 성공 ≠ 권한 충분"** 이고, **읽기는 되는데 쓰기만 실패하면 십중팔구 코드가 아니라 리소스 공유 설정**이다.

---

## 배운 점

- **단방향 동기화는 반대 방향에 조용한 데이터 유실 구멍을 남긴다.** 원본이 한쪽이면, 다른 쪽에서 생긴 변경은 다음 동기화 때 덮여 사라진다. 양방향이 필요한지 처음에 따져봐야 한다.
- **동기화 도구의 제1원칙은 "남의 데이터를 망치지 않기".** append-only로 범위를 좁히니, 자동화를 믿고 맡길 수 있었다. 똑똑하게 병합하려다 번역을 덮어쓰는 것보다 백 배 낫다.
- **자동화 트리거는 "사람의 기억"이 아니라 "이벤트"에 건다.** 키를 추가하는 행위(push) 자체가 동기화를 부르게 하면, 누구도 잊을 수 없다.
- **권한 에러는 계층을 나눠 본다.** 인증 / OAuth 스코프 / 리소스 공유 — 셋은 다른 층이다. "읽기 성공 + 쓰기 403"이라는 신호만으로 범위를 리소스 공유로 좁힐 수 있었다.
- **시크릿은 로컬엔 파일, CI엔 시크릿 저장소.** 같은 키를 환경에 따라 다른 방식으로 주입한다는 감각.

---

> 작은 스크립트 하나였지만, "데이터가 어디서 와서 어디로 가는가"를 그림으로 그려보는 것만으로 문제의 정체(단방향 구멍)와 해법(반대 방향 + append-only)이 또렷해졌다.
